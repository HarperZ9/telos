## Mission Posture

Project Telos is no longer scoped as only an accountability or agent-ops
project. It is a frontier R&D operating system for proof-carrying work across
software, science, learning, creative systems, advanced computation, and
high-stakes technical domains.

The posture is broad, but the evidence discipline stays narrow:

- A domain is allowed into scope when it can be represented as source intake,
  workspace context, a falsifiable claim, a bounded experiment, a receipt, and
  a verification gate.
- A claim is public only when its supporting packet says exactly what was
  measured, what was not measured, what failed, and what remains unverifiable.
- Military-adjacent, defense-adjacent, nuclear, biochem, cybernetics, AI/ML,
  quant, finance, robotics, medicine, and advanced computing domains are in
  scope as frontier R&D lanes.
- Public artifacts must not turn high-risk source material into operational
  instructions. High-risk technical payloads stay internal, summarized,
  redacted, or reframed as safety, measurement, simulation, assurance, or
  publication candidates until a stronger review gate exists.

## Strategic Thesis

Most tools in the market solve one layer: intake, search, notebooks,
observability, orchestration, proof assistants, experiment tracking,
compiler/runtime receipts, or publication. Telos should win where those layers
must be joined:

```text
source -> context -> route -> action -> measurement -> verdict -> receipt -> paper/product
```

The differentiator is not "another agent framework" or "another scientific
language." The differentiator is a proof-centered substrate where every
frontier claim can be traced from evidence to execution to verification.

## Frontier Domain Map

The domain map is intentionally wide. Each field needs a public-safe first
packet, a hard falsification gate, and a private/internal boundary for raw or
high-risk material.

| Domain | First honest packet | What must be measured | What must not be claimed yet |
| --- | --- | --- | --- |
| Quantum computing | Stabilizer/QEC packet, Clifford equivalence, resource-estimation ledger. | Code distance, error model, syndrome table, decoder behavior, resource assumptions. | Hardware QEC, quantum advantage, cryptographic security, or fault tolerance without primary evidence. |
| Physics and mathematics | Formal/numerical proof packet, Lean/BuildLang paired replay, theorem-search provenance. | Axioms, discretization, invariant, solver config, negative controls, formal proof boundary. | Solved grand problem without independent proof review and reproducible artifacts. |
| Biology and medicine | Causal/microscopy/lab measurement packet with source and instrument receipts. | Sample prep, instrument state, calibration, units, image/video extraction, causal assumptions. | Clinical efficacy, diagnosis, wet-lab safety, or mechanistic truth from source summaries alone. |
| Robotics and cybernetics | Sim-to-real packet with plant model, controller, safety envelope, latency, and measured trace. | Units, controller law, state estimate, command log, environment, failures, safety stop. | Real-world safety, surgical validity, autonomy guarantee, or deployment readiness from toy fixtures. |
| Nuclear and energy | Industrial science packet with public primary sources, units, model assumptions, regulation state, and replayed calculations. | Source provenance, physics assumptions, factory state, regulatory claims, safety boundaries, uncertainty. | Safety, throughput, licensing, enrichment, weapons, or plant performance claims without primary source and expert review. |
| AI/ML research | Model-foundry packet with dataset boundary, evals, objective monitor, compute ledger, and contamination controls. | Dataset lineage, eval definition, budget, action trace, verifier outcome, residual risk. | Frontier-lab pretraining capacity or benchmark superiority without reproduced official rules. |
| Quant and finance | Backtest/strategy receipt with data source, split rules, costs, slippage, baseline, and live/paper boundary. | Data version, feature lag, transaction costs, trial count, regime stress, drawdown. | Investment advice, capital allocation, or expected return. |
| Security and assurance | Defensive proof packet for scan, patch, release, redaction, provenance, or agent audit. | Source, issue, patch diff, tests, exploitability boundary, release effect, secret scan. | Operational misuse instructions, credential acquisition, or unauthorized target action. |
| Color/rendering/scientific compute | Color/render measurement proof kit with ICC/ACES/OCIO-style inputs and measured outputs. | Display state, transforms, gamut, delta-E, renderer config, artifact hashes, frame budget. | Universal color correctness or production-grade calibration without device/instrument evidence. |
| Education and functional learning | Learning Forge prooflesson with source receipts, exercises, falsification, and transfer checks. | Source refs, learner task, answer evidence, misconception traps, retention/replay. | Credential or assessment automation beyond explicitly allowed logistics and owned labs. |

| `deprecated-lineage` | Historical name or repo preserved for lineage only. | `LINEAGE.md`, registry notes |

### Publication Gates

1. Source gate: every public claim points to source receipts or primary docs.
2. Scope gate: every paper states what it does not prove.
3. Falsification gate: every result has at least one negative fixture or explicit failure condition.
4. Reproduction gate: runnable artifacts have commands and tests.
5. Risk gate: high-risk details are summarized or internal until a publication review says otherwise.
6. Market gate: market-position claims remain hypotheses unless backed by a comparison matrix.
7. Freshness gate: Index and Forum are rerun before project-level claims.

### Phase 1: Make Paper Candidates Real

Target: 1-2 weeks.

- Convert current official/whitepaper packets into a publication queue:
  causal workbench, BuildLang scientific runtime, embodied sim-to-real, QEC,
  biology/hyphal routing, TI Morse industrial science, and formal PDE replay.
- Treat `docs/research/PUBLICATION-QUEUE-2026-07-02.md` as the source of truth
  for paper priority until a newer queue supersedes it.
- For each paper candidate, define:
  title, contribution, primary evidence, methods, negative controls, what is
  not claimed, related work, demo command, and publication target.
- Create an arXiv/ORCiD-ready style guide for Telos papers:
  no unsupported superlatives, no "solved" language without external proof,
  citations separated from source leads, exact artifact hashes when available.

  than grand-problem claims until independent reviewers and stronger artifacts
  exist.

## Decision Rules

- If a claim cannot be measured, call it a hypothesis.
- If a source is a video transcript, call it a source lead until primary
  evidence is captured.
- If a packet is high-risk, publish the receipt shape and non-claims before
  technical details.
- If BuildLang/buildc can express the computation, prefer a Build receipt over
  a handwritten JavaScript fixture.
- If a domain has no negative control, it is not ready for a public "proof"
  label.
- If a doc disagrees with a receipt, the receipt wins and the doc is stale.
- If a repo is dirty, do not treat its local docs as shipped until committed or
  explicitly marked working-tree evidence.
